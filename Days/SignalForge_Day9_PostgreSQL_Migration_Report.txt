SignalForge Day 9 PostgreSQL Migration Report
Date: June 28, 2026

Day 9 goal:
Move telemetry events and incidents from in-memory storage to PostgreSQL using SQLAlchemy models, indexes, and clean database access patterns.


1. What Changed

The backend storage layer was completely rewritten from an in-memory dictionary to a SQLAlchemy-backed database layer.

Before Day 9:
- Events stored in a Python list.
- Incidents stored in a Python list.
- Seen event IDs tracked in a Python set.
- Open incident keys tracked in a Python set.
- Recent events by service tracked in a Python deque.
- All data lost on backend restart.

After Day 9:
- Events stored in a PostgreSQL (or SQLite) table.
- Incidents stored in a PostgreSQL (or SQLite) table.
- Duplicate detection via primary key constraint on event_id.
- Open incident detection via SQL query on status != 'resolved'.
- Recent events via SQL query with ORDER BY timestamp DESC LIMIT 500.
- Data survives backend restarts.

New files:
- signalforge_mvp/backend/app/database.py
- signalforge_mvp/backend/app/models.py

Updated files:

3. Database Access Layer

File: signalforge_mvp/backend/app/storage.py

DatabaseStore class provides the same interface as the old InMemoryStore:

- add_event(event) -> bool: checks PK for duplicates, inserts if new
- get_recent_events(tenant_id, service_name) -> list: SQL query with DESC LIMIT 500
- add_incident(incident) -> bool: checks for open incident via SQL, inserts if none
- get_incident(incident_id) -> Incident | None: PK lookup
- update_incident_status(incident_id, status, timeline_entry) -> Incident | None: PK update
- list_incidents() -> list: ORDER BY created_at DESC
- list_events(limit=50) -> list: ORDER BY timestamp DESC LIMIT
- reset() -> None: truncates both tables (for tests)

Session management: each method creates its own SQLAlchemy session with context manager, commits on success.

6. JSON Serialization Fix

Timeline entries contain datetime timestamps. SQLite's JSON column cannot serialize datetime objects natively.

Fix: Use Pydantic's model_dump(mode="json") when storing timeline entries in the JSON column. This converts datetime to ISO strings. When reading back, Pydantic automatically parses ISO strings back to datetime objects.

This fix applies to both add_incident and update_incident_status.

9. Beginner-Friendly Explanation

Before Day 9, SignalForge stored everything in its brain (computer memory). When the backend restarted, it forgot everything.

After Day 9, SignalForge stores everything in a database (like a notebook). When the backend restarts, it opens the notebook and remembers everything.

The database has two tables:
- telemetry_events: every request, log, trace, and deployment event
- incidents: every incident with its evidence and timeline

The database also has indexes (like bookmarks in the notebook) that make it fast to find:
- Events for a specific service
- Events in a specific time range
- Incidents by status
- Open incidents for a service

This is how real production systems work. Databases survive restarts, crashes, and reboots.
