SignalForge — Day 25 Report
API Key Auth and Tenant Isolation

What was built today
--------------------
Added API key authentication and tenant isolation across the entire backend.
All endpoints (except health check) now require an X-API-Key header. The
tenant_id from the authenticated key is enforced on every data query, preventing
cross-tenant data leakage. The frontend sends the API key on every request.

New files
- backend/app/auth.py
  Simple auth module with `get_current_tenant` dependency:
  - Validates `X-API-Key` header is present
  - Maps API keys to tenant IDs via in-memory dictionary
  - Returns 401 Unauthorized with clear message on missing or invalid key
  - For MVP: `sf-api-key-demo` → `demo-company`, `sf-test-key` → `demo-company`

- backend/tests/test_auth.py
  3 tests verifying auth behavior:
  - Missing API key → 401
  - Invalid API key → 401
  - Valid API key → data returned

Modified files (backend routers — all now enforce auth + tenant isolation)
- backend/app/routers/ingest.py
  - `Depends(get_current_tenant)` added to ingest endpoint

    cross-tenant data injection)

What the auth system proves
---------------------------
- Every API endpoint (except health) requires authentication
- The tenant_id from the API key is enforced on all data queries
- A tenant cannot read, update, or delete another tenant's incidents or runbooks
- The ingest endpoint cannot be used to inject data into another tenant's namespace
- The frontend handles 401 errors gracefully with a user-friendly message

Interview talking points
------------------------
- Why API key auth instead of JWT? "For an MVP, API keys are simpler to implement
  and test. JWT would be the next step for production with token refresh, scopes,
  and OAuth2 integration. The architecture is designed to swap `get_current_tenant`
  for a JWT dependency without changing any router code."
- How do you prevent cross-tenant data leakage? "Every storage method accepts an
  optional tenant_id parameter. When the API layer passes the authenticated tenant_id,
  the storage layer filters or rejects records that don't match. This is defense in
  depth: even if a router bug misses a check, the storage layer enforces the boundary."
- Why override tenant_id on ingest? "The payload could contain any tenant_id. By
  overriding it with the authenticated tenant from the API key, we prevent a malicious
  client from injecting events into another tenant's namespace."
- How would you scale this to production? "Replace the in-memory API_KEY_TENANTS
  dictionary with a database lookup or a secrets manager (AWS Secrets Manager,
  HashiCorp Vault). Add rate limiting per API key. Add audit logging for auth
  failures."

Resume bullets
--------------
- Implemented API key authentication (`X-API-Key`) with tenant isolation across
  all backend endpoints using FastAPI dependency injection
- Enforced tenant_id boundaries at the storage layer, preventing cross-tenant data
  leakage for incidents, events, runbooks, and search queries
- Updated frontend to send authenticated headers and handle 401 errors gracefully
- Added 4 auth-specific tests verifying 401 responses on missing/invalid keys and
