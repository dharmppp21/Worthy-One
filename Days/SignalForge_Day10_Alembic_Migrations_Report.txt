SignalForge Day 10 Alembic Migrations Report
Date: June 29, 2026

Day 10 goal:
Add database migrations with Alembic and replace create_all-style setup with a proper migration workflow so schema changes are version-controlled, repeatable, and documented.


1. What Changed

Before Day 10:
- Tables were created with Base.metadata.create_all() on every backend startup.
- No way to track schema changes over time.
- No rollback capability.
- No version-controlled history of database schema evolution.
- Adding a new column or index required manual SQL or dropping tables.

After Day 10:
- Alembic manages schema versions in `backend/alembic/versions/`.
- `app/main.py` runs `alembic upgrade head` on startup instead of `create_all()`.
- Every schema change generates a migration script that can be committed to git.
- Migrations can be upgraded, downgraded, and inspected.
- The same migration runs on SQLite (dev) and PostgreSQL (prod).

4. How Migrations Run on Startup

File: signalforge_mvp/backend/app/main.py

- create_app() now calls run_migrations() before registering routers.
- run_migrations() builds an Alembic Config pointing to alembic.ini.
- It overrides sqlalchemy.url with DATABASE_URL from app.database.
- It calls command.upgrade(alembic_cfg, "head").
- If the database is already at the latest version, this is a no-op.
- If the database is empty, the migration creates all tables and indexes.

This means:
- First-time setup: start Docker, start backend, tables are created automatically.
- Schema change: create a new migration, restart backend, new migration is applied.
- No manual SQL required.

Downgrade one revision:

python -m alembic downgrade -1

7. Beginner-Friendly Explanation Of What Happened

Before Day 10, the backend used a simple but risky approach: every time it started, it ran SQL CREATE TABLE statements for every model. This is fine for a quick demo, but it has problems in production:

- If you add a new column, create_all() does not add it to an existing table.
- If you rename a column, create_all() does not rename it.
- If you add an index, create_all() might recreate the whole table.
- There is no history of what changed and when.

Alembic solves this by treating schema changes like code changes. Each change is a migration script with two parts: upgrade (apply the change) and downgrade (undo the change). You generate a migration with --autogenerate, review it, and apply it. The scripts are stored in version control so every developer and every deployment environment runs the same schema evolution.

In SignalForge, the backend now runs alembic upgrade head on startup. This means:

- A new developer clones the repo, starts the backend, and the database is ready.
- A developer adds a field to IncidentModel, generates a migration, and the next backend restart applies it.
- A staging deployment receives the new migration script and applies it automatically.
- If something goes wrong, you can downgrade to a known-good state.


8. What Day 10 Proves

Day 10 proves that SignalForge can manage schema evolution like a production system:

1. Schema is version-controlled alongside application code.
2. New environments get the correct schema automatically on startup.
3. Schema changes are reviewable, reversible, and repeatable.
4. The same migration works on SQLite (local dev) and PostgreSQL (production).
5. The backend does not rely on risky create_all() behavior.

This is the foundation for future schema changes: adding runbook tables, deployment history tables, event log partitions, or any other feature that needs new columns or indexes.
