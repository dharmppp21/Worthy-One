SignalForge — Day 29 Report
Demo Seed Data and Walkthrough Guide

What was built today
--------------------
Created a deterministic demo seed script and a comprehensive walkthrough guide
that tells a complete, repeatable story: healthy traffic → bad deployment →
cascading failure → incident creation → structured evidence → root cause →
runbook retrieval → AI triage. The demo runs in 5 minutes and requires no
manual setup or waiting for events to process.

New files
- scripts/seed_demo.py
  Deterministic seed script that creates a complete demo dataset directly via
  the EventProcessor and storage layer:
  - Phase 1: 20 healthy metric events across all 5 services (200ms, 200 OK)
  - Phase 2: 1 deployment event for notification-service v2.1.0
  - Phase 3: 20 bad metric events for notification-service (500, 2000-2500ms)
  - Phase 4: 10 bad trace events showing checkout-service -> notification-service failures
  - Phase 5: 10 bad metric events for checkout-service (500, 2200ms)
  - Phase 6: 5 error log events ("connection pool exhausted")
  - Phase 7: 2 runbooks (checkout-service and notification-service playbooks)
  - Result: 50 events, 1 critical incident, 2 runbooks, 1 deployment

  The script runs in ~2 seconds, prints a narrative summary, and shows the
  incident timeline with deployment correlation evidence.

- DEMO.md
  8-step demo walkthrough with:
  - Setup instructions (30 seconds: run seed script, start backend, open dashboard)
  - The story narrative (notification-service v2.1.0 bug causes cascade)
  - 8 numbered steps with exact API commands and expected responses:
    1. Show healthy baseline (/events)
    2. Show bad deployment (/deployments)
    3. Show incident with evidence (/incidents)
    4. Show service graph (/graph)
    5. Show root cause analysis (/services/{name}/root-cause)
    6. Show runbook retrieval (/runbooks)
    7. Show AI triage (/incidents/{id}/ai-triage)
    8. Show search (/search?q=notification)
  - Talking points for each step (what to say in an interview)
  - Closing statement (15-second summary)
  - Commands cheat sheet
  - Demo tips (run seed first, tell the story not the tech, mention the numbers)

What the demo proves
--------------------
- **Repeatability:** Run the seed script, start the backend, and the full story
  is already there. No waiting, no randomness, no "let me see if it works this time."
- **Completeness:** The demo covers every major feature: ingest, anomaly detection,
  incident creation, deployment correlation, evidence timeline, service graph,
  root cause scoring, runbook retrieval, AI triage, and search.
- **Determinism:** The seed script uses the EventProcessor directly, so the same
  events always produce the same incident with the same evidence. This is crucial
  for a demo — you can't have a flaky story in an interview.
- **Speed:** The entire demo story is created in ~2 seconds. The backend doesn't
  need to be running during seeding. Then you start the backend and the data
  is immediately available.
- **Narrative flow:** The 8 steps build on each other: baseline → change →
  failure → detection → dependency → cause → fix → AI analysis. Each step
  answers the question the previous step raised.
