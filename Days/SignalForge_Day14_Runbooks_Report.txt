2. Backend: Runbook CRUD API

File: signalforge_mvp/backend/app/routers/runbooks.py

Endpoints:
- POST /runbooks — creates a new runbook linked to a service
- GET /runbooks — lists all runbooks, optionally filtered by tenant_id and service_name
- GET /runbooks/{id} — returns a single runbook
- PATCH /runbooks/{id} — updates title, description, or steps
- DELETE /runbooks/{id} — deletes a runbook

File: signalforge_mvp/backend/app/storage.py

Methods added to DatabaseStore:
- create_runbook() — inserts a new runbook, returns False if ID already exists
- get_runbook() — fetches by ID
- list_runbooks() — lists with optional tenant_id and service_name filters
- update_runbook() — PATCH-style partial update
- delete_runbook() — removes by ID
- get_runbook_for_service() — returns the most recent runbook for a service (for future incident linking)


3. Database Schema

File: signalforge_mvp/backend/app/models.py

New RunbookModel table:
- id: String(128), primary key
- tenant_id: String(128), nullable=False
- service_name: String(128), nullable=False, indexed
- title: String(512), nullable=False
- description: String(2048), nullable=False
- steps: JSON, nullable=False (default empty list)
- created_at: DateTime(timezone=True), indexed
- updated_at: DateTime(timezone=True)

Index: idx_runbooks_tenant_service on (tenant_id, service_name)

Why this schema:
- service_name is indexed because the primary query pattern is "get runbook for service X".
- steps is JSON so the order is preserved and steps can be added/removed without schema changes.
- title and description are sized for human-readable content.
- tenant_id allows multi-tenant separation even though the MVP is single-tenant.

7. Beginner-Friendly Explanation Of What Happened

Before Day 14, when a service failed, you had to figure out how to fix it from scratch or ask someone who knew. There was no place to write down "when this service breaks, do these steps."

Day 14 adds runbooks — like recipe cards for fixing services. Here's how it works:

1. You create a runbook for "notification-service" with steps like "Check SMTP config" and "Verify queue depth."
2. The runbook is stored in the database and linked to notification-service.
3. When notification-service fails in the future, you (or a teammate) can open the Runbooks tab and see exactly what to do.
4. You can update the runbook as procedures change, or delete it if it's no longer needed.

This is operational memory — knowledge that persists beyond any single person. The system now remembers not just what happened (incidents) and what changed (deployments), but also how to fix things (runbooks).
