2. New Event ID Behavior

Every telemetry event now has:

event_id

Example:

sample-metric-payment-001

Why this matters:

In real distributed systems, clients often retry requests. Without event IDs, the same telemetry event could be processed twice. That could incorrectly increase error counts, distort latency windows, or create duplicate incidents.

Current behavior:

- If client sends event_id, SignalForge uses it.
- If client does not send event_id, the API generates one.
- If the same event_id is sent again, SignalForge returns duplicate=true.
- Duplicate events are not stored again.
- Duplicate events are not analyzed again.
- Duplicate events cannot create duplicate incident side effects.

Example first response:

{
  "accepted": true,
  "event_id": "sample-metric-payment-001",
  "duplicate": false
}

Example duplicate response:

{
  "accepted": true,
  "event_id": "sample-metric-payment-001",
  "duplicate": true
}


3. Validation Rules Added

Validation lives in:

signalforge_mvp/backend/app/schemas.py

Current rules:

Metric events:

- metric events named http_request require attributes.status_code.
- metric events named http_request require attributes.latency_ms.
- status_code must be an integer between 100 and 599.
- latency_ms must be a non-negative number.

Log events:

- log events require a non-empty message.
- severity must be one of:
  - debug
  - info
  - warning
  - error
  - critical

Trace events:

- trace events require trace_id.
- trace events named service_call require attributes.caller.
- trace events named service_call require attributes.callee.

Deployment events:

- deployment events named service_deployed require attributes.version.

Extra unknown top-level fields are rejected.

Invalid payloads return FastAPI's normal:

422 Unprocessable Entity


4. Duplicate Protection Implementation

Duplicate protection lives in:

signalforge_mvp/backend/app/storage.py

The in-memory store now tracks:

seen_event_ids

When a new event arrives:

1. Check whether event.event_id already exists.
2. If yes, return false and do not store/process the event again.
3. If no, record the event_id, store the event, and update the recent service window.

The service layer lives in:

signalforge_mvp/backend/app/services/telemetry_service.py

It now returns duplicate=true when storage rejects a duplicate event.


5. Simulator Updated

The simulator now sends event_id values.

Updated file:

signalforge_mvp/simulator/traffic_simulator.py

It now creates IDs such as:

- metric-<uuid>
- log-<uuid>
- deployment-<uuid>

It also sends log events when simulated HTTP requests fail.

This makes the simulator feel more like real production telemetry:

- metric event: request status and latency
- log event: error message when a request fails
- deployment event: bad payment-service version deployment

9. Beginner-Friendly Explanation

Before Day 3, SignalForge accepted events but did not know whether a repeated event was a retry or a new event.

Now every event carries an ID.

That lets the backend say:

"I have already processed this exact event, so I will not count it again."

This is important because real systems retry all the time. Without duplicate protection, retrying telemetry could create fake error spikes.

Day 3 also made the event schema smarter:

- Request metrics must include status code and latency.
- Logs must include a message.
- Traces must include a trace ID.
- Deployments must include a version.

So SignalForge now rejects incomplete telemetry instead of silently accepting bad data.
