SignalForge Day 12 Service Graph Report
Date: July 1, 2026

Day 12 goal:
Add service graph support from trace events, including backend endpoint and frontend visualization of service dependencies.


1. What Changed

Before Day 12:
- Trace events with name="service_call" were ingested and stored, but never analyzed.
- No way to see which service calls which.
- The dashboard only showed incident cards, with no topology context.
- When checkout-service failed, there was no visual indication it depends on notification-service.

After Day 12:
- Backend exposes GET /graph that returns nodes (services) and edges (caller -> callee) extracted from trace events.
- Frontend dashboard has a tab toggle: "Incidents" and "Service Graph".
- The Service Graph view renders an SVG topology diagram with services in a circle and curved arrows showing call dependencies.
- Edge thickness scales with call count. Edge labels show count when > 1.
- The graph updates every 5 seconds via react-query polling.

2. Backend: GET /graph Endpoint

File: signalforge_mvp/backend/app/routers/graph.py

The endpoint:
- Calls store.get_service_graph()
- Returns ServiceGraphResponse with nodes and edges
- No query parameters required

File: signalforge_mvp/backend/app/storage.py

The get_service_graph() method:
- Queries telemetry_events where event_type="trace" AND name="service_call"
- For each row, extracts attributes.caller and attributes.callee
- Builds a set of unique node IDs (all services)
- Builds edge_counts dictionary keyed by (source, target) tuple
- Returns ServiceGraphResponse with sorted nodes and edges

Why this approach:
- The simulator already emits trace events with caller/callee (Day 6).
- No schema migration needed — we query existing data.
- Aggregation happens at query time, so the graph reflects all historical traces.
- If a service has no trace events, it won't appear in the graph (until it emits one).


3. Frontend: Service Graph Visualization

File: signalforge_mvp/frontend/src/components/ServiceGraph.tsx

The component:
- Receives nodes and edges as props
- Arranges nodes in a circular layout using trigonometry
- Renders SVG with:
  - <circle> for each service node (r=32, blue border, light fill)
  - <text> for service name labels centered in each node
  - <path> for edges with quadratic Bezier curves (curved arrows)
  - <marker> for arrowheads at end of each path
  - <text> for edge count labels at curve midpoint
- Edge stroke width scales with call count (min 1px, max 4px)
- Returns empty state when no nodes exist

File: signalforge_mvp/frontend/src/App.tsx

Changes:
- Added activeTab state: "incidents" | "graph"
- Added tab navigation bar with two buttons
- Incidents tab: existing incident grid and detail panel
- Graph tab: ServiceGraphView component with data from fetchServiceGraph
- Graph query uses refetchInterval: 5000 (5 seconds)
- Graph query only runs when graph tab is active (enabled flag)

Why this approach:
- Circular layout is deterministic and easy to understand.
- No external graph library needed — pure SVG.
- Tab navigation keeps the UI clean and focused.
- Conditional query fetching avoids unnecessary API calls when the user is on the incidents tab.

6. Beginner-Friendly Explanation Of What Happened

When you have a microservice system, services call each other. For example, checkout-service calls payment-service to charge a customer. The simulator already records these calls as "trace events" with caller and callee fields. Before Day 12, these events were stored but never visualized.

Day 12 adds a way to see the big picture: which services talk to which. The backend queries all trace events, counts how many times each pair of services called each other, and returns a graph structure. The frontend draws this as a circle of services with arrows between them.

Why this matters: When notification-service fails, the graph immediately shows that checkout-service depends on it. This is root-cause analysis without reading log files — the topology makes the dependency visible.


7. What Day 12 Proves

Day 12 proves that SignalForge can:

1. Extract meaningful topology from existing telemetry data.
2. Aggregate trace events into a directed graph without schema changes.
3. Render the graph in the frontend with pure SVG (no heavy dependencies).
4. Switch between incident view and topology view in the same dashboard.
5. Poll for graph updates without impacting the incidents view.

This is a building block for future features: highlighting failing services in the graph, coloring edges by error rate, or showing latency heatmaps on the topology.
