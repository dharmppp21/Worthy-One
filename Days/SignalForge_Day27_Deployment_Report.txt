Why these decisions
-------------------
- **Frontend multi-stage build:** Building in the image guarantees the build
  environment matches the runtime environment. No "works on my machine" issues.
- **Nginx for static serving:** Nginx is faster than Node.js for serving static
  files. Gzip and cache headers reduce bandwidth. SPA routing handles React Router.
- **Simulator as a profile:** The simulator is a one-off traffic generator. Using
  `--profile simulator` means it doesn't start by default. You run it when you want
  to generate test traffic.
- **Frontend proxies /api/* to backend:** This avoids CORS issues in production
  (same origin). The browser sends requests to `http://localhost/api/...` and nginx
  forwards them to the backend container. This is the standard production pattern.
- **AWS architecture document:** For interviews, you need to be able to draw the
  architecture on a whiteboard and justify each choice. The document gives you
  all the talking points. It also shows you understand cost, security, and scaling.

Interview talking points
------------------------
- Why ECS Fargate over Lambda? "Lambda is great for short-lived tasks. But our
  backend has WebSocket connections, background Kafka consumers, and in-memory
  state. Lambda has a 15-minute max duration and no native WebSocket support.
  ECS Fargate is the right abstraction for a long-running service."
- How do you scale from 100 to 10,000 RPS? "1. Scale backend tasks: ECS target
  tracking scales from 3 to 20 tasks based on CPU. 2. Scale RDS: move from
  db.t3.medium to db.r6g.xlarge, enable read replicas. 3. Scale Redis: ElastiCache
  cluster mode adds shards. 4. Scale Kafka: MSK Serverless auto-scales partitions.
  5. Add caching: CloudFront for static assets, API response caching. 6. Database
  connection pooling: RDS Proxy handles 1000+ connections. Each step is independent."
- How do you handle zero-downtime deployments? "ECS rolling updates: start new
  tasks, wait for ALB health checks, drain old tasks. Alembic migrations run
  as a one-off ECS task before deployment, with a lock to prevent concurrent
  migrations."
- What if the database goes down? "RDS Multi-AZ has automatic failover in 60-120
  seconds. The backend health check fails, ALB stops routing traffic, and ECS
  keeps old tasks running. Kafka retains messages for 7 days, so no events are
  lost. Workers resume from their last committed offset when the DB recovers."
- Why not EKS? "EKS is Kubernetes — powerful but complex. For a Python backend
  with 3-20 tasks, Fargate is simpler: no control plane, no node management, no
  kubectl. If we needed complex service mesh or custom networking, EKS would be
  the next step."
- How do you handle secrets? "The 12-factor app says config in env vars. But AWS
  best practice is Secrets Manager for sensitive values. In ECS, you reference a
  secret by ARN and the agent injects it at runtime. The container never sees the
  ARN, just the value. For local development, we use .env files (never committed)."
- Cost estimate: "Dev stack is ~$130/month (small RDS, single Redis node, 1-3
  ECS tasks). Prod stack is ~$400/month (Multi-AZ RDS, 3-shard Redis cluster,
  3-20 ECS tasks, MSK Serverless). Reserved instances save 30-40%."
