2. Current behavior / architecture details

Embedding service (three-tier fallback):
1. If `OPENAI_API_KEY` env var is set, uses OpenAI `text-embedding-3-small` (1536 dims).
2. If `sentence_transformers` is installed, uses `all-MiniLM-L6-v2` (384 dims, auto-padded to 1536 for storage compatibility).
3. Otherwise, `is_available()` returns `False` and all embedding operations are skipped.

pgvector integration:
- The `embeddings` table has columns: `id`, `entity_type` ("incident"/"runbook"), `entity_id`, `embedding` (vector(1536)), `created_at`.
- Unique constraint on `(entity_type, entity_id)` for UPSERT support.
- The Alembic migration checks `bind.dialect.name == "postgresql"` before creating anything. On SQLite, the migration is a no-op.
- Cosine similarity search uses `1 - (embedding <=> query_embedding)` where `<=>` is pgvector's cosine distance operator.

Search endpoint behavior:
- `GET /search?q=checkout` → keyword search (same as Day 15).
- `GET /search?q=checkout&semantic=true` → tries semantic search. If embedding service is unavailable or no embeddings exist, falls back to keyword search. No error, no broken UI.

Automatic embedding generation:
- When an incident is created (via anomaly detection), `incident_engine.py` embeds `title + summary` and stores it.
- When a runbook is created (via API), `runbooks.py` embeds `title + description + steps` and stores it.
- Both only run if `embedding_service.is_available()` is True.

5. Beginner-Friendly Explanation

Keyword search (what we built on Day 15) is like using Ctrl+F — it only finds exact words. If you search "slow" but the incident says "latency degradation," keyword search won't find it.

Semantic search is smarter. It converts text into a "vector" — a list of numbers that captures the meaning of the words. "Slow checkout" and "checkout latency spike" have very similar vectors because they mean similar things. pgvector is a PostgreSQL extension that can compare these vectors quickly and find the most similar ones.

The system tries three ways to generate these vectors:
1. OpenAI's API (if you have a key)
2. A free local model (if you installed it)
3. If neither works, it just uses keyword search instead — no broken features, no errors.

When an incident or runbook is created, the system automatically generates its vector and stores it. Later, when you search with semantic mode on, your query is also converted to a vector and the database finds the closest matches.
