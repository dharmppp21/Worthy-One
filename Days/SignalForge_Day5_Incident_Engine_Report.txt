2. Duplicate Open Incident Prevention

Logic lives in:

signalforge_mvp/backend/app/storage.py

How it works:

- InMemoryStore.open_incident_keys tracks (tenant_id, service_name) pairs that have open incidents.
- When incident_engine calls store.add_incident(incident), the store checks if the service already has an open incident.
- If yes, the incident is rejected and not created again.
- If no, the incident is stored and the key is added to open_incident_keys.

This prevents duplicate incident spam when the same service keeps failing.

4. Incident Status Lifecycle

SignalForge supports three statuses:

- investigating: default when an incident is created. The service is still open.
- mitigated: a partial fix has been applied. The service is still open.
- resolved: the incident is closed and the service key is removed from open_incident_keys.

API endpoint:

PATCH /incidents/{incident_id}/status

Request body:

{
  "status": "mitigated",
  "actor": "on-call-engineer",
  "note": "Rolled back payment-service v42."
}

What happens internally:

1. If the new status is the same as the current status, the update is a no-op and no timeline entry is added.
2. If the status changes, a status_changed timeline entry is appended with actor, timestamp, and note.
3. updated_at is refreshed.
4. If the new status is resolved, the (tenant, service) key is removed from open_incident_keys, allowing future incidents for that service.

8. Beginner-Friendly Explanation

Imagine SignalForge creates an incident for payment-service.

Without duplicate protection:

Every bad request could create a new incident. You might end up with 50 identical incidents about the same service. That is noisy and useless.

With duplicate protection:

SignalForge says: "I already know payment-service is broken. I will not create another incident until the first one is resolved."

The incident timeline is like a logbook:

- Entry 1: "Incident opened at 10:00 AM because error rate was 35%."
- Entry 2: "Evidence attached: 20 recent requests, average latency 1496 ms."
- Entry 3: "Status changed to mitigated at 10:15 AM by on-call engineer. Note: rolled back v42."
- Entry 4: "Status changed to resolved at 10:30 AM."

When the incident is resolved, SignalForge can create a new incident for payment-service if it breaks again. This is a complete incident lifecycle.
