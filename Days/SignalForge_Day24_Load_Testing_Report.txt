SignalForge — Day 24 Report
Load Testing: Measured Throughput, Detection Delay, and API Latency

What was built today
--------------------
Added load testing scripts that measure real performance numbers against a
running backend. The scripts are self-contained: they start the backend,
run the tests, and output a structured JSON report. This gives concrete
numbers for the resume and reveals the exact bottleneck.

New files
- backend/tests/load/locustfile.py
  Locust load test with two user classes:
  - TelemetryUser: simulates services sending metrics (10× weight for healthy
    metrics, 3× for errors, 2× for traces, 1× for logs). Also polls read
    endpoints (incidents, events, graph, health, search, runbooks).
  - ReadOnlyUser: simulates dashboard users polling data every 1–3 seconds.

- backend/tests/load/run_load_tests.py
  Orchestrator script that:
  1. Starts uvicorn backend in a subprocess (SQLite file mode, dummy Redis)
  2. Waits for /health to return 200
  3. Measures API latency for all 6 read endpoints (10 samples each)
  4. Measures ingest throughput with a 10-second burst (20 concurrent workers)
  5. Measures incident detection delay (20 sequential bad events → poll until
     incident appears)
  6. Measures fixed-count concurrent ingest (1000 requests, 50 workers)
  7. Stops backend, saves JSON report to load_test_results.json

- backend/tests/load/debug_detection.py
  Isolated test script that sends 20 bad events to a fresh service and verifies
  incident detection works. Used to debug why the initial detection test failed.

- backend/requirements-locust.txt
  locust>=2.44,<3.0 for load testing (separate from main requirements to keep
  production dependencies clean).

- backend/tests/load/load_test_results.json
  Generated JSON report with all measured numbers.

Bug found during testing
------------------------
The initial incident detection test failed (no incident detected). The debug
script revealed that event IDs 0–9 were being rejected with 422 validation
errors because event_id="debug-0" is only 7 characters, but the TelemetryEvent
schema requires min_length=8. Only events 10–19 were stored, giving 10 events
instead of the required 20.

Fix: Changed event ID generation to use zero-padding (e.g., debug-0000,
load-00000001) so all IDs exceed 8 characters. After the fix, all 20 events
were accepted and the incident was detected in 784 ms.

Bottleneck analysis
-------------------
The bottleneck is SQLite concurrent writes, not the API or anomaly detection.

At 20 concurrent workers: 0 errors, p99=522 ms. This is acceptable for a
development laptop.

At 50 concurrent workers: 0.4% errors, p99=3,361 ms. The error messages are
"database is locked" — SQLite's single-writer model cannot handle 50
concurrent write transactions.

The read endpoints are unaffected (all under 30 ms p95) because SQLite handles
concurrent reads well. The issue is purely write contention.

Upgrade path: Switch to PostgreSQL (already in the Docker Compose stack).
PostgreSQL handles concurrent writes with row-level locking and connection pooling.
On the same hardware, PostgreSQL would likely scale to 500+ RPS.

Interview talking points
------------------------
- What did load testing reveal? "On SQLite, the backend handles ~48 RPS with
  0 errors at 20 concurrent workers. At 50 workers, p99 latency jumps to 3.3s
  and 0.4% of requests fail with 'database is locked.' This proves SQLite is
  the bottleneck, not the API or anomaly detection logic. With PostgreSQL
  (already in the Docker stack), I'd expect 500+ RPS."
- Why measure detection delay? "It's not enough to say 'we detect anomalies.'
  You need to know how fast. 784 ms from the 20th bad event to incident
  creation is a concrete number I can put on a resume and discuss in an
  interview."
- How did you find the bug? "The initial detection test failed. I wrote a debug
  script that printed each event's status code. I saw 422 errors for events
  0–9 but 202 for 10–19. The error message said 'string_too_short' for
  event_id. The schema requires min_length=8, and 'debug-0' is only 7 chars.
  I padded the IDs and the test passed. This is exactly how I'd debug in
  production: isolate, reproduce, inspect, fix."
- How would you scale this? "The API itself is stateless. The bottleneck is
  the database. With PostgreSQL + connection pooling, I expect 500+ RPS on a
  single instance. With Kafka + multiple worker instances, I can scale
  horizontally to thousands of events per second."
