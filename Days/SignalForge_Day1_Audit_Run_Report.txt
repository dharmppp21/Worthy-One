SignalForge Day 1 Audit And Run Report
Date: June 20, 2026

Day 1 goal:
Audit the existing SignalForge starter project, verify the backend API and simulator, fix immediate startup issues, and confirm that bad payment-service traffic creates the first incident.


1. Current Project Structure Confirmed

The current starter project lives in:

signalforge_mvp/

It has three main parts:

1. Backend
   Path: signalforge_mvp/backend
   Purpose: FastAPI API that accepts telemetry events and creates incidents.

2. Simulator
   Path: signalforge_mvp/simulator
   Purpose: Python script that sends fake service traffic to the backend.

3. Frontend
   Path: signalforge_mvp/frontend
   Purpose: React dashboard shell that can show incidents from the backend.


2. Current Backend Architecture In Simple Terms

The backend is the "brain" of the current MVP.

Important files:

- app/main.py
  Defines the HTTP API endpoints:
  - GET /health
  - POST /ingest
  - GET /events
  - GET /incidents

- app/schemas.py
  Defines the shape of telemetry events and incidents using Pydantic.

- app/storage.py
  Stores events and incidents in memory for now.

- app/anomaly.py
  Checks recent request events and decides whether a service looks unhealthy.

- app/incident_engine.py
  Creates an incident when anomaly.py says something is wrong.

Simple flow:

Service event -> POST /ingest -> store event -> analyze recent events -> maybe create incident -> view incident at /incidents


3. Current Simulator Architecture In Simple Terms

The simulator is a fake production system.

It pretends to be multiple backend services:

- checkout-service
- payment-service
- inventory-service

At first it sends mostly healthy traffic.
After 30 events, it simulates a bad payment-service deployment.
Then it sends slow and error-heavy payment-service events.

That bad traffic is what causes SignalForge to create the first incident.


4. Current Frontend Structure Confirmed

The frontend is a Vite + React + TypeScript app.

Important files:

- frontend/src/App.tsx
  Main dashboard UI.

- frontend/src/api.ts
  Calls the backend /incidents endpoint.

- frontend/src/types.ts
  TypeScript incident type.

- frontend/src/App.css
  Dashboard styling.

For Day 1, I confirmed the frontend structure. The required Day 1 runtime verification was focused on backend plus simulator because those are the parts needed to create the first incident.


5. Startup Issue Found And Fixed

Issue:
The backend requirements originally pinned old versions:

- fastapi==0.115.0
- uvicorn==0.30.6
- pydantic==2.8.2

This failed on the available local Python version:

Python 3.14.3

The old pydantic-core dependency did not have a ready Python 3.14 wheel, so pip tried to build from source and failed.

Fix:
I updated backend requirements to compatible version ranges:

- fastapi>=0.116,<1.0
- uvicorn[standard]>=0.35,<1.0
- pydantic>=2.11,<3.0

Installed result:

- fastapi 0.138.0
- uvicorn 0.49.0
- pydantic 2.13.4
- pydantic-core 2.46.4 with Python 3.14 wheel

I also added a README note to set:

$env:PYTHONUTF8='1'

Reason:
The workspace path contains Unicode characters, so Python tooling on Windows is safer in UTF-8 mode.

8. How To Analyze The MVP Yourself

Step 1: Open the backend docs

http://127.0.0.1:8000/docs

This shows all FastAPI endpoints.

Step 2: Watch events

Open:

http://127.0.0.1:8000/events

This shows recent telemetry events received by the backend.

Look for:

- service_name
- event_type
- status_code
- latency_ms

Step 3: Watch incidents

Open:

http://127.0.0.1:8000/incidents

This shows detected incidents.

Look for:

- which service failed
- severity
- evidence
- created_at

Step 4: Read anomaly.py

Path:

signalforge_mvp/backend/app/anomaly.py

This file explains the current detection rule:

- Wait until there are at least 20 request events for a service.
- Check the last 20 request events.
- If error rate is at least 20 percent, flag anomaly.
- If average latency is at least 1000 ms, flag anomaly.

Step 5: Read traffic_simulator.py

Path:

signalforge_mvp/simulator/traffic_simulator.py

This file explains why an incident happens:

- First 30 events are mostly normal.
- Then the simulator sends a deployment event for payment-service v42.
- Then payment-service starts returning more failures and high latency.


9. Beginner-Friendly Explanation Of What Happened

Imagine payment-service is a real backend service in an e-commerce app.

Normally:

- It responds quickly.
- Most requests succeed.

After the simulated bad deployment:

- Many requests become slow.
- Some requests fail with status code 500.

SignalForge receives those request events through /ingest.
It stores recent events in memory.
It checks the last 20 request events for payment-service.
It sees the error rate is too high.
Then it creates an incident.

That is the first working version of the project.

In one sentence:

SignalForge watches fake production traffic, notices payment-service is becoming unhealthy, and creates an incident with evidence.


10. What Day 1 Proves

Day 1 proves the core loop works:

1. Services can send telemetry.
2. Backend can receive telemetry.
3. Recent telemetry can be analyzed.
4. An anomaly can become an incident.
5. You can inspect the incident through an API.

This is the foundation. Future days will replace simple in-memory pieces with production-grade pieces such as PostgreSQL, Redis, Kafka or Redpanda, runbooks, vector search, AI triage, and a stronger dashboard.
