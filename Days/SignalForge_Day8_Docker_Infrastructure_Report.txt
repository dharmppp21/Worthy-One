2. Docker Compose Services

File: signalforge_mvp/docker-compose.yml

PostgreSQL (postgres:16-alpine):
- Port: 5432
- Credentials: signforge / signforge
- Database: signforge
- Named volume: postgres_data (survives container restarts)
- Healthcheck: pg_isready every 5 seconds
- Restart policy: unless-stopped

Redis (redis:7-alpine):
- Port: 6379
- Named volume: redis_data (survives container restarts)
- Append-only mode enabled for durability
- Healthcheck: redis-cli ping every 5 seconds
- Restart policy: unless-stopped

Both services are isolated in Docker containers and expose ports on localhost so the backend can connect to them.


3. Backend Dockerfile

File: signalforge_mvp/backend/Dockerfile

- Base image: python:3.12-slim
- Installs system dependencies for asyncpg (gcc, libpq-dev)
- Copies requirements.txt and installs Python dependencies
- Copies app/ directory
- Exposes port 8000
- Runs uvicorn with host 0.0.0.0

This is ready for production deployment but not yet included in docker-compose.yml. The backend is still run locally via uvicorn --reload for development.


4. Environment Variables

File: signalforge_mvp/backend/.env.example

| Variable | Default | Purpose |
|----------|---------|---------|
| DATABASE_URL | postgresql+asyncpg://signforge:signforge@localhost:5432/signforge | PostgreSQL connection (Day 9) |
| REDIS_URL | redis://localhost:6379/0 | Redis connection (Day 11) |
| LOG_LEVEL | INFO | Logging verbosity (Day 26) |
| ENVIRONMENT | development | Environment flag (Day 26) |

These are documented but not yet consumed by the backend. The backend still uses in-memory storage.


5. Python Dependencies Added

Updated file: signalforge_mvp/backend/requirements.txt

Added:
- sqlalchemy[asyncio]>=2.0,<3.0 — SQLAlchemy ORM with async support
- asyncpg>=0.29,<1.0 — Async PostgreSQL driver
- redis>=5.0,<6.0 — Redis client

These are installed in the virtual environment but the backend code does not use them yet. The migration from in-memory to PostgreSQL happens on Day 9. Redis rolling windows happen on Day 11.

7. How to Stop Infrastructure

docker-compose down

Remove volumes (wipes data):
docker-compose down -v


8. Why This Matters for Interviews

Before Day 8, the project only ran on in-memory storage. That was fine for development but not production-ready.

After Day 8, the project has:
- A reproducible local infrastructure stack (PostgreSQL + Redis)
- Named Docker volumes for data persistence
- Health checks on both services
- A Dockerfile ready for containerized deployment
- Environment variables documented for different environments

This is a real DevOps setup that hiring managers will recognize. You can explain:
- "PostgreSQL is the durable source of truth."
- "Redis is the hot operational state for fast rolling window checks."
- "Docker Compose makes the local environment reproducible across machines."


9. Beginner-Friendly Explanation

Imagine SignalForge is a restaurant. Before Day 8, the kitchen (backend) wrote all orders on a whiteboard (in-memory). If the kitchen closed, the orders were gone.

After Day 8:
- The restaurant installed a permanent filing cabinet (PostgreSQL) for all order records.
- The restaurant installed a quick-access board (Redis) for the chef to see today's hot orders without opening the filing cabinet.
- The filing cabinet and the quick-access board are set up in the back room (Docker containers) and stay there even when the kitchen closes.
- The kitchen hasn't started using the filing cabinet yet (that's Day 9). The whiteboard is still there. But the filing cabinet is ready.
