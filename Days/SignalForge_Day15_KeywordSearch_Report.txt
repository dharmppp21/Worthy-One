2. Current behavior / architecture details

Search backend:
- GET /search?q={keyword} performs case-insensitive ILIKE search across incident title, summary, service_name and runbook title, description, service_name.
- Results are capped at 50 per table and merged into a single list sorted by created_at descending.
- Each result has a type field ("incident" or "runbook") so the frontend can render them differently.

Search frontend:
- New "Search" tab (🔍) alongside Incidents, Service Graph, and Runbooks.
- Search input with submit button. Results appear as cards in the same grid layout as incidents.
- Incident results are clickable and open the detail panel (switches back to Incidents tab).
- Runbook results are read-only cards with purple left border.

Incident detail memory:
- When viewing an incident detail, a "Related Runbooks" section fetches runbooks for that incident's service_name.
- Shows title, description, and numbered steps. If no runbooks exist, the section is hidden.
- This connects operational knowledge (runbooks) directly to incident context without manual lookup.

5. Beginner-Friendly Explanation

Before Day 15, if you wanted to find an old incident or a runbook, you had to scroll through lists manually. Now you can type a keyword like "checkout" and the system searches both incidents and runbooks at once, showing results from both categories.

When you open an incident, the dashboard also looks up any runbooks written for that same service and shows them right there. This means if checkout-service has an incident and someone already wrote a runbook for it, you see the fix steps immediately without switching tabs.

The search is basic keyword matching (SQL LIKE), not AI/vector search. But it works as a solid foundation — later we can upgrade to semantic search.
