SignalForge Day 2 Backend Refactor Report
Date: June 21, 2026

Day 2 goal:
Clean up the FastAPI backend structure into routers and services, add helpful beginner-friendly comments, document every endpoint in the README, and verify the MVP still runs exactly as before.


1. What Changed

Before Day 2:

Most backend endpoint logic lived directly inside:

signalforge_mvp/backend/app/main.py

That was okay for Day 1, but it would become hard to maintain once the project adds PostgreSQL, Redis, Kafka or Redpanda, service graph, runbooks, and AI triage.

After Day 2:

The backend is split into clearer layers:

signalforge_mvp/backend/app/
  main.py
  routers/
    health.py
    ingest.py
    events.py
    incidents.py
  services/
    telemetry_service.py
  schemas.py
  storage.py
  anomaly.py
  incident_engine.py


2. What Each Layer Means

main.py:
Creates the FastAPI app, configures CORS, and connects route modules to the app.

routers/:
Contains HTTP endpoint definitions.
Routers should answer the question:
"What URL exists, what input does it accept, and what service function should handle it?"

services/:
Contains business workflow logic.
Services should answer the question:
"What should the application actually do after an API request arrives?"

storage.py:
Acts like the temporary in-memory database for the MVP.
Later this will be replaced or backed by PostgreSQL and Redis.

anomaly.py:
Decides whether recent service traffic is unhealthy.

incident_engine.py:
Turns an anomaly into an incident.

schemas.py:
Defines the data shapes for events, responses, and incidents.


3. Why This Refactor Matters

This makes the backend easier to extend.

For example:

- Day 3 can improve telemetry schemas without touching route wiring too much.
- Day 4 can improve anomaly detection inside anomaly.py.
- Day 8 and Day 9 can add PostgreSQL/Redis by changing storage and services.
- Later Kafka/Redpanda workers can reuse the service workflow.

The main idea:

Routes stay thin.
Services coordinate work.
Storage stores data.
Anomaly detection decides health.
Incident engine creates incidents.

8. How To Analyze The Refactor Yourself

Start from:

signalforge_mvp/backend/app/main.py

You will see that it imports routers and includes them in the app.

Then open:

signalforge_mvp/backend/app/routers/ingest.py

You will see that /ingest accepts a TelemetryEvent and calls:

ingest_telemetry_event(event)

Then open:

signalforge_mvp/backend/app/services/telemetry_service.py

That is the main Day 2 workflow file.

It does this:

1. Generate an event_id.
2. Store the event.
3. Load recent events for the same tenant and service.
4. Run anomaly detection.
5. Maybe create an incident.
6. Return accepted=true and the event_id.

Then open:

signalforge_mvp/backend/app/anomaly.py

That file decides whether the recent service window is unhealthy.

Then open:

signalforge_mvp/backend/app/incident_engine.py

That file creates the incident object if anomaly.is_anomaly is true.


9. Beginner-Friendly Mental Model

Before Day 2, the backend was like one notebook page with everything written together.

After Day 2, the backend is like a small company:

- main.py is reception: it opens the office and connects departments.
- routers are front-desk counters: each counter handles one type of request.
- services are managers: they coordinate the actual workflow.
- storage is the filing cabinet.
- anomaly.py is the health inspector.
- incident_engine.py writes the incident report.

The behavior did not change. The organization changed so the project can grow without becoming messy.
