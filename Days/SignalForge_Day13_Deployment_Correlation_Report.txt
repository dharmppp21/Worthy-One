SignalForge Day 13 Deployment Correlation Report
Date: July 2, 2026

Day 13 goal:
Add deployment event tracking and correlate incidents with recent deployments in the incident evidence, enabling change-based root cause analysis.


1. What Changed

Before Day 13:
- Deployment events were ingested and stored, but never linked to incidents.
- When notification-service failed after a bad deploy (v42), the incident only showed anomaly stats (error rate, latency).
- Engineers had to manually check if a deployment happened around the same time.
- No "What changed?" context in the incident timeline.

After Day 13:
- `storage.get_recent_deployments()` queries deployment events for a service within a configurable window (default 30 minutes).
- `incident_engine.maybe_create_incident()` calls this method when creating an incident.
- If a recent deployment exists, the incident timeline gets an `evidence_added` entry with:
  - Deployment version (e.g., v42)
  - Deployment timestamp
  - Number of deployments in the window
- The frontend renders this timeline entry with a yellow highlight and a rocket badge (🚀 v42).
- `GET /deployments` endpoint lists all recent deployment events.

2. Backend: get_recent_deployments()

File: signalforge_mvp/backend/app/storage.py

The method:
- Takes tenant_id, service_name, and window_minutes (default 30)
- Builds a cutoff timestamp: now - window_minutes
- Queries PostgreSQL for events where:
  - event_type = "deployment"
  - service_name matches
  - timestamp >= cutoff
- Orders by timestamp DESC (most recent first)
- Returns list of TelemetryEvent objects

Why 30 minutes:
- Most deployment-related failures surface within minutes (config errors, missing env vars, incompatible DB schemas).
- 30 minutes is a reasonable default. Can be adjusted per use case.
- The window is configurable, so future features could allow per-service overrides.


3. Backend: Incident Engine Correlation

File: signalforge_mvp/backend/app/incident_engine.py

The flow when an anomaly triggers incident creation:
1. Anomaly detected for service X.
2. Incident engine calls `store.get_recent_deployments(tenant_id, service_name, 30)`.
3. If deployments found:
   - Take the most recent one (index 0, since DESC order).
   - Extract version from attributes.
   - Create timeline entry with deployment metadata.
4. Incident is created with the deployment evidence in its timeline.

The timeline entry looks like:
{
  "timestamp": "2026-06-28T12:05:00Z",
  "event_type": "evidence_added",
  "message": "Recent deployment detected: notification-service v42 at 2026-06-28T12:00:00Z.",
  "metadata": {
    "deployment_event_id": "deploy-notification-001",
    "deployment_version": "v42",
    "deployment_time": "2026-06-28T12:00:00Z",
    "deployments_in_window": 1
  }
}

This is the first step toward "change-based root cause analysis." The system now automatically suggests that a deployment might be related to the incident.

5. API: GET /deployments

File: signalforge_mvp/backend/app/routers/deployments.py

Endpoint: GET /deployments

Returns recent deployment events from the database.

Example response:
{
  "deployments": [
    {
      "event_id": "deploy-notification-001",
      "tenant_id": "tenant-a",
      "service_name": "notification-service",
      "event_type": "deployment",
      "timestamp": "2026-06-28T12:00:00Z",
      "name": "service_deployed",
      "attributes": { "version": "v42" }
    }
  ]
}

Use this endpoint to audit deployment history or build a deployment timeline view.

7. Beginner-Friendly Explanation Of What Happened

When something breaks, the first question engineers ask is: "What changed?"

Before Day 13, SignalForge detected failures and created incidents, but you had to manually check if a deployment happened recently. Now the system does that automatically.

Here's how it works:
1. The simulator deploys notification-service v42 (a bad version).
2. The simulator starts sending failing requests to notification-service.
3. SignalForge detects the anomaly and creates an incident.
4. At the moment of incident creation, the system checks: "Did this service have a deployment recently?"
5. It finds the v42 deployment from 5 minutes ago.
6. It adds this to the incident timeline: "Recent deployment detected: notification-service v42."
7. When you open the dashboard, you see a yellow-highlighted timeline entry with a rocket badge.

This is powerful because it immediately points to a likely root cause. Instead of wondering why the service failed, you see: "Oh, v42 was deployed 5 minutes before the failure. That's probably the cause."


8. What Day 13 Proves

Day 13 proves that SignalForge can:

1. Query historical deployment events from the durable database.
2. Correlate incidents with deployments at creation time.
3. Add structured deployment metadata to the incident timeline.
4. Render deployment correlation visually in the frontend (badge + highlight).
5. Expose deployment history via a dedicated API endpoint.

This is a building block for future features:
- Deployment rollback recommendations ("v42 caused 3 incidents, consider rolling back to v41").
- Change-based alerting (alert the team that deployed v42, not just the on-call engineer).
- Deployment health gates (block deployment if the previous version caused incidents).

The key principle: "What changed?" is the most important question in incident response. SignalForge now answers it automatically.
