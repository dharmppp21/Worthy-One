2. Current Rolling Window Policy

SignalForge currently analyzes only:

event_type = metric
name = http_request

It ignores logs, traces, and deployment events for anomaly detection for now.
Those will become useful later for root-cause evidence.

Current rolling window:

- Minimum request samples before detection: 20
- Window size analyzed: latest 20 request events for the same tenant and service

Why this matters:

The detector should not create incidents from only 1 or 2 unlucky requests.
It waits until it has enough recent traffic to make a more stable decision.


3. Current Warning Thresholds

A service becomes warning if any of these are true:

- Error rate >= 20%
- Average latency >= 1000 ms
- p95 latency >= 1500 ms

Example:

If 4 out of the last 20 requests fail:

4 / 20 = 20%

That becomes a warning anomaly.


4. Current Critical Thresholds

A service becomes critical if any of these are true:

- Error rate >= 50%
- Average latency >= 1800 ms
- p95 latency >= 2500 ms

Example:

If 10 out of the last 20 requests fail:

10 / 20 = 50%

That becomes a critical anomaly.

11. Beginner-Friendly Explanation

Imagine SignalForge is watching payment-service.

It does not panic after one failed request.

Instead, it waits until it has 20 recent request events.

Then it asks:

1. How many of these 20 requests failed?
2. How slow were they on average?
3. What was the p95 latency, meaning how slow were the slowest important requests?

If too many requests failed, it creates an incident.
If requests became too slow, it creates an incident.
If failures or latency are very bad, it marks the incident critical.

In simple words:

SignalForge now says not just "something is wrong," but "something is wrong because error rate, average latency, or p95 latency crossed this exact threshold."
