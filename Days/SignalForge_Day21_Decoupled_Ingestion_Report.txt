SignalForge — Day 21 Report
Decoupled Ingestion: API Publishes, Worker Processes

What was built today
--------------------
Refactored the ingestion pipeline so the API accepts events quickly and returns
immediately, while worker processors handle the heavy lifting of database
writes, Redis hot-state updates, anomaly detection, and incident creation.

New file
- backend/app/services/event_processor.py
  Worker-owned pipeline class with a single .process() method.
  Stages: 1) store in PostgreSQL + push to Redis, 2) anomaly detection,
  3) incident creation. The API never calls this directly.

Modified files
- backend/app/routers/ingest.py
  POST /ingest now returns 202 Accepted when the event is published to the
  Kafka stream (mode="async"). When Kafka is unavailable it falls back to
  synchronous processing via EventProcessor (mode="sync").

- backend/app/services/telemetry_service.py
  ingest_telemetry_event() is now a thin sync-fallback wrapper that delegates
  to EventProcessor.process(). All query/list operations remain here.

- backend/app/services/kafka_consumer_worker.py
  The background worker now calls event_processor.process() directly instead
  of going through the telemetry_service wrapper. Updated docstring to explain
  backpressure and horizontal scaling.

- backend/app/schemas.py
  IngestResponse.mode comment updated to include "async" as a valid value.

- backend/app/redis_client.py
  Widened exception handling in RedisWindowStore.__init__ to catch URL parsing
  errors (e.g. invalid port numbers) so Redis gracefully degrades instead of
  crashing on import.

- backend/tests/conftest.py
  Changed dummy Redis URL from port 99999 (out of range, causes ValueError)
  to port 12345 (valid range, connection refused gracefully).

- backend/tests/test_event_processor.py
  6 new tests covering: duplicate detection, healthy events create no incident,
  anomaly events create an incident, single open incident per service, DB
  storage, Redis hot-state storage.

Architecture decisions
----------------------
- API and processing are decoupled. The FastAPI endpoint only validates and
  publishes; the worker owns the full pipeline. This is the standard pattern
  for high-throughput ingestion systems.
- Backpressure: if workers are slow, the Kafka queue grows. The API stays
  fast because it never waits for processing. Consumers pull at their own
  pace.
- Horizontal scaling: run N worker processes with the same group_id. Kafka
  automatically rebalances partitions across them. More workers = more
  throughput, without changing the API code.
- Sync fallback: when Kafka is unavailable (local dev, Docker not running),
  the API processes inline and returns mode="sync". Zero-downtime graceful
  degradation.

Interview talking points
------------------------
- Why 202 Accepted? "202 signals the client that the event was accepted but
  processing is asynchronous. The client doesn't need to wait for anomaly
  detection or incident creation."
- Why a separate EventProcessor class? "It makes the ownership boundary
  explicit. The API publishes; the worker processes. In an interview, this
  shows I understand separation of concerns and can explain backpressure
  and horizontal scaling."
- What happens if the worker crashes? "Kafka retains messages for a configured
  retention period. When the worker restarts, it resumes from its last committed
  offset. Unprocessed events are not lost."
- How would you scale this? "Run more worker containers. Kafka rebalances
  topic partitions across consumers in the same group automatically. No code
  changes needed."
