2. Current behavior / architecture details

Event-driven ingestion flow:
1. Simulator or client sends `POST /ingest` with a telemetry event.
2. If Kafka is available: the API serializes the event to JSON and publishes it to the `telemetry_events` topic. Returns `accepted: true, mode: "kafka"` immediately.
3. The Kafka consumer worker (background thread) consumes the event from the topic.
4. The worker processes the event: stores in DB, runs anomaly detection, creates incident if needed, broadcasts via WebSocket.
5. If processing fails, the worker retries up to 3 times with exponential backoff (1s, 2s, 4s).
6. If all retries fail, the event is sent to the `telemetry_events_dead_letter` topic and logged.
7. If Kafka is unavailable: the API falls back to synchronous processing (same as Days 1-19).

Fallback behavior:
- Kafka unavailable: `ingest_telemetry_event` is called synchronously, same behavior as before.
- Redis unavailable: anomaly detection reads from DB instead of Redis (Day 11 behavior).
- Both unavailable: the system still works, just without streaming and hot rolling windows.

5. Beginner-Friendly Explanation

Before Day 20, when you sent a telemetry event to the API, it was processed immediately and stored in the database. This is called synchronous processing. It works but has a problem: if the API is getting lots of events, it can get overwhelmed because every event is processed right away.

Kafka is a message queue (a "message broker"). Instead of processing the event immediately, the API puts it in a queue (called a "topic") and says "someone else will process this later." Then a separate worker reads from the queue and processes the events one by one.

This is like the difference between a restaurant where the chef cooks each order as soon as it's placed (synchronous) vs one where orders are written on a ticket and the chef works through them in order (event-driven). The second approach is more scalable.

Here's the flow:
1. You send an event to the API.
2. The API puts it in the Kafka queue (topic: `telemetry_events`).
3. The API immediately returns "accepted" to you.
4. A background worker reads from the queue.
5. The worker stores the event, checks for anomalies, creates incidents if needed.
6. If the worker fails to process an event, it tries again (up to 3 times, waiting longer each time).
7. If it still fails after 3 tries, the event goes to a "dead letter" queue where someone can investigate later.

If Kafka is not running, the API falls back to the old synchronous behavior. So the system works either way.
