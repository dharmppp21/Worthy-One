Project Idea: SignalForge - AI Incident Triage Platform for Distributed Microservices

One-line resume pitch:
Build a scalable observability and incident triage platform that ingests logs, traces, metrics, deployment events, and runbooks from microservices, detects abnormal behavior in real time, retrieves similar past incidents using vector search, and generates ranked root-cause hypotheses with remediation steps.


1. The Core Intuition

Modern products are not one big backend anymore. A single user action can pass through an API gateway, authentication service, payment service, recommendation service, database, cache, message queue, and third-party API. When something goes wrong, the problem is rarely "we have no data." The real problem is that the data is scattered.

Logs tell one part of the story.
Metrics tell another part.
Traces show request flow.
Deployments show what changed.
Runbooks explain what humans usually do.
Past incidents contain historical knowledge.

The intuition behind this project is:

"Can I build a system that acts like an intelligent on-call assistant for distributed systems by connecting all these signals together?"

Instead of making engineers manually jump between dashboards, logs, traces, Slack messages, and documentation, SignalForge collects the signals, correlates them, detects abnormal patterns, searches past incidents, and suggests likely root causes.

In an interview, explain it like this:

"I wanted to solve a real SRE problem: reducing mean time to detect and resolve incidents in microservice systems. In production, failures are noisy. A latency spike in one service may actually be caused by a downstream database, a bad deployment, a cache miss storm, or a queue backlog. So I designed SignalForge as an event-driven observability and AI triage platform. It ingests telemetry from services, builds a dependency graph from traces, performs streaming anomaly detection, retrieves similar historical incidents using embeddings, and ranks possible root causes with evidence."


2. Why This Is Real-World Useful

Companies like Google, Meta, Amazon, Netflix, Uber, and large SaaS companies operate thousands of services. Their reliability depends on how quickly engineers can detect, understand, and fix incidents.

This project is useful because it targets a real operational pain:

- On-call engineers waste time correlating logs, metrics, traces, and deployment history.
- Teams need faster incident diagnosis during outages.
- Junior engineers often do not know which dashboard, runbook, or service owner to check first.
- Repeated incidents are common, but previous knowledge is often buried in documents or ticket threads.
- AI is useful here only if it is grounded in real system data, not if it behaves like a generic chatbot.

SignalForge helps by providing:

- A live incident dashboard.
- A service dependency graph.
- Anomaly alerts based on real-time telemetry.
- Semantic search over previous incidents and runbooks.
- Root-cause suggestions with supporting evidence.
- Suggested next actions for engineers.

This is not a toy CRUD app. It is a real platform engineering project.

4. Main Product Features

Feature 1: Telemetry Ingestion

Each demo microservice sends logs, traces, metrics, and deployment events into SignalForge.

Examples:

- HTTP request latency.
- Error rate.
- CPU and memory usage.
- Database query latency.
- Queue lag.
- Cache hit rate.
- Deployment version.
- Trace spans across services.

Implementation idea:

- Build an ingestion API using FastAPI.
- Accept JSON events from services.
- Validate events using Pydantic.
- Add tenant_id, service_name, trace_id, timestamp, and event_type.
- Push events into Kafka or Redpanda for asynchronous processing.

Interview explanation:

"I did not write directly to the database from the ingestion API because ingestion can spike during incidents. I used a message broker so the API can accept events quickly and the processing layer can scale independently."


Feature 2: Stream Processing And Anomaly Detection

A worker consumes telemetry events and maintains rolling windows.

Examples:

- Error rate increased from 1 percent to 18 percent in 5 minutes.
- p95 latency doubled after a deployment.
- Queue lag is growing continuously.
- A downstream dependency is failing for multiple upstream services.

Implementation idea:

- Use Redis sorted sets or time-window counters for hot rolling metrics.
- Use simple anomaly rules first: z-score, moving average, threshold breach, rate-of-change.
- Later add ML models such as Isolation Forest or forecasting.
- Emit incident_candidate events when anomalies are detected.

Interview explanation:

"I started with explainable anomaly detection instead of jumping directly to black-box ML. In production incident response, engineers need to understand why an alert fired. A moving average or z-score with evidence is easier to trust."


Feature 3: Service Dependency Graph

Use traces to infer which services call each other.

Example:

checkout-api -> payment-service -> fraud-service -> postgres
checkout-api -> inventory-service -> redis

Implementation idea:

- Parse trace spans.
- Store parent-child service relationships.
- Maintain service health status.
- Show a React graph where unhealthy services are highlighted.

Interview explanation:

"A dependency graph helps avoid blaming the first failing service. If five services show errors but they all depend on the same downstream database, the graph helps identify the common dependency."


Feature 4: Incident Memory With Vector Search

Store past incidents, postmortems, runbooks, alerts, and fixes.

Examples:

- "Redis cache eviction caused login latency."
- "Payment-service timeout after fraud API rate limit."
- "Database connection pool exhaustion after deployment."

Implementation idea:

- Store incident text and metadata in PostgreSQL.
- Generate embeddings for incident summaries and runbooks.
- Use pgvector for semantic similarity search.
- Retrieve similar incidents when a new incident occurs.

Interview explanation:

"When an incident happens, I do not want the AI to answer from general knowledge. I want it to retrieve company-specific memory: past incidents, runbooks, service ownership, and recent changes. That is why I used retrieval-augmented generation."


Feature 5: Root-Cause Ranking Engine

Given an incident candidate, rank possible causes.

Signals used:

- Recent deployments.
- Error rate changes.
- Latency changes.
- Dependency graph blast radius.
- Similar past incidents.
- Service ownership.
- Logs with repeated error signatures.

Example output:

Likely root cause:
1. payment-service deployment v42 introduced timeout errors.
Evidence:
- Error rate increased 3 minutes after deployment.
- p95 latency increased from 120 ms to 1.8 s.
- Similar incident found from March 2026 involving the same dependency.
- Logs show repeated "connection timeout to fraud-service".

Interview explanation:

"I made the AI output evidence-backed hypotheses, not final truth. The system ranks likely causes and shows why it thinks each one matters. This keeps the engineer in control."


Feature 6: Incident Dashboard

Build a React dashboard with:

- Live incidents.
- Service dependency graph.
- Timeline of events.
- Top anomalies.
- Retrieved similar incidents.
- Root-cause hypotheses.
- Suggested next actions.
- Incident status: investigating, mitigated, resolved.

This gives the project a strong demo. A recruiter or interviewer can immediately understand what it does.


5. High-Level Architecture

Frontend:

- React + TypeScript.
- Dashboard for incidents, service graph, timelines, and search.

API Layer:

- FastAPI gateway.
- Auth, tenant isolation, incident APIs, search APIs, WebSocket updates.

Ingestion Layer:

- FastAPI ingestion service.
- Validates telemetry events.
- Writes to Kafka or Redpanda.

Streaming Layer:

- Worker consumers process logs, traces, metrics, and deployments.
- Maintains rolling windows.
- Detects anomalies.
- Emits incident candidates.

Storage Layer:

- PostgreSQL for incidents, services, traces, deployments, users, teams, and runbooks.
- pgvector for semantic search over incidents and runbooks.
- Redis for hot counters, rate limits, WebSocket fanout, and short-lived incident state.

AI Layer:

- Embedding worker for runbooks and incident documents.
- Retrieval service for similar incidents.
- LLM reasoning service for root-cause hypothesis generation.
- Guardrails to force answers to cite retrieved evidence.

Infrastructure:

- Docker Compose for local development.
- AWS ECS or EKS for deployment.
- RDS PostgreSQL.
- ElastiCache Redis.
- MSK or self-hosted Redpanda/Kafka.
- CloudWatch or Prometheus/Grafana for monitoring.


6. Data Flow

1. A microservice receives user traffic.
2. It emits logs, metrics, traces, and deployment metadata.
3. SignalForge ingestion API receives the event.
4. The event is pushed into Kafka/Redpanda.
5. Stream workers consume events and update rolling windows in Redis.
6. Durable event summaries are written into PostgreSQL.
7. The anomaly detector identifies suspicious behavior.
8. The incident engine creates an incident candidate.
9. The retrieval service searches similar incidents and runbooks using pgvector.
10. The AI reasoning service generates ranked hypotheses with evidence.
11. The dashboard updates in real time through WebSockets.
12. The engineer marks the incident as resolved and writes a postmortem.
13. The postmortem becomes future incident memory.


7. Scalability And System Design Choices

Ingestion scalability:

- Use async FastAPI endpoints.
- Batch writes to Kafka.
- Apply rate limits per tenant.
- Return quickly after event validation.
- Avoid database writes in the hot ingestion path.

Stream processing scalability:

- Partition topics by tenant_id and service_name.
- Use consumer groups to scale workers horizontally.
- Use idempotency keys to handle duplicate events.
- Use retries and dead-letter queues for malformed events.

Database scalability:

- Partition high-volume tables by time and tenant.
- Store raw events separately from summarized incident data.
- Index by tenant_id, service_name, timestamp, trace_id, and incident_id.
- Archive old telemetry data to object storage.

Vector search scalability:

- Store embeddings in pgvector.
- Use metadata filters such as service_name, tenant_id, severity, and date range.
- Use approximate indexes for faster similarity search.

Real-time dashboard scalability:

- Use Redis pub/sub or streams for live updates.
- WebSocket gateway subscribes only to incidents relevant to the user.
- Send summarized deltas instead of raw telemetry.

Reliability:

- Make ingestion at-least-once.
- Make processors idempotent.
- Use dead-letter queues.
- Add health checks and retries.
- Preserve incident summaries even if AI generation fails.

AI reliability:

- Never let the AI be the source of truth.
- Show retrieved documents and telemetry evidence.
- Force structured JSON output.
- Add confidence levels.
- Let the user inspect why a hypothesis was ranked.

13. Why This Project Beats Generic AI Projects

Many AI resume projects are wrappers around an LLM API. This idea is stronger because the AI is only one part of a larger distributed system.

The real engineering value comes from:

- Ingesting high-volume data.
- Processing streams.
- Modeling incidents.
- Correlating multiple signal types.
- Designing reliable storage.
- Building a useful operator dashboard.
- Making AI grounded, explainable, and safe.

In an interview, this lets you say:

"The AI is not the product by itself. The product is the incident intelligence system. The AI layer becomes useful because it is grounded in telemetry, dependency graphs, deployment history, and previous incidents."

That is a much more mature engineering story.

15. Final Interview-Style Explanation

"SignalForge is an AI-assisted incident triage platform for distributed microservices. I built it because real production outages are not caused by a lack of monitoring data, but by the difficulty of correlating too many signals quickly. The system ingests logs, metrics, traces, and deployment events through a scalable event pipeline. Stream workers maintain rolling health windows and detect anomalies. Traces are used to build a service dependency graph, so the system can understand blast radius and downstream failures. When an incident appears, the platform retrieves similar past incidents and runbooks using vector search, then generates ranked root-cause hypotheses with evidence. The dashboard gives engineers a timeline, affected services, likely causes, and suggested next actions. The main design challenge was making the system scalable and trustworthy: ingestion is decoupled through Kafka, processors are idempotent, hot state is stored in Redis, durable data is stored in PostgreSQL, and the AI is grounded in retrieved evidence instead of making unsupported claims."
