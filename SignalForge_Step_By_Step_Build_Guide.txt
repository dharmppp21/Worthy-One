SignalForge Step-By-Step Build Guide
Beginner-friendly explanation + phase-by-phase implementation plan


0. First Understand The Project In Simple Words

Imagine you built an online shopping app.

It has many backend services:

- checkout-service: handles "Place Order"
- payment-service: charges money
- inventory-service: checks stock
- fraud-service: checks whether payment is suspicious
- notification-service: sends email or SMS
- database: stores orders
- redis: stores cache

Now imagine users suddenly cannot place orders.

The company is losing money every minute.

The engineer on-call has to answer:

- Which service is broken?
- Did a recent deployment cause it?
- Is the database slow?
- Is Redis failing?
- Is payment-service timing out?
- Did this happen before?
- What should we do now?

In real companies, engineers use logs, metrics, traces, dashboards, Slack messages, and runbooks to answer these questions.

SignalForge is a project that helps answer these questions automatically.

Think of SignalForge as:

"A smart doctor for backend systems."

The backend services are the patient.
Logs, metrics, and traces are symptoms.
Past incidents are medical history.
Runbooks are treatment instructions.
SignalForge collects symptoms, detects abnormal behavior, remembers similar past problems, and suggests likely causes.


1. The Main Idea In One Practical Example

Normal situation:

- checkout-service receives 100 requests per minute.
- payment-service usually replies in 100 ms.
- error rate is usually 1 percent.

Bad situation:

- A new payment-service version is deployed.
- payment-service starts timing out when calling fraud-service.
- checkout-service becomes slow.
- users see failed orders.
- logs show "fraud-service timeout".
- error rate jumps from 1 percent to 18 percent.

SignalForge should detect:

"Something is wrong. Payment-service was deployed 3 minutes before errors started. Fraud-service timeout logs are increasing. A similar incident happened before. Top suggested action: rollback payment-service or check fraud-service timeout configuration."

That is the project.


2. What You Are Actually Building

You will build these parts:

1. Demo microservices
   Fake checkout/payment/inventory services that generate realistic logs, metrics, traces, and errors.

2. Telemetry ingestion API
   A FastAPI backend that receives events from services.

3. Event processing worker
   A worker that reads events and calculates health signals.

4. Anomaly detector
   Code that detects when error rate or latency becomes abnormal.

5. Incident engine
   Code that creates an incident when anomalies are found.

6. Incident memory
   A database of previous incidents and runbooks.

7. AI triage
   A service that retrieves similar past incidents and generates root-cause hypotheses.

8. Dashboard
   A React UI showing live incidents, affected services, timelines, and suggested causes.

9. Production polish
   Docker, tests, load testing, auth, deployment, monitoring, and a strong README.

5. Phase 0: Setup And Mental Model

Goal:

Understand the data moving through the system before writing complex code.

The most important object in the project is a telemetry event.

A telemetry event means:

"A small piece of information emitted by a service."

Examples:

- checkout-service had one successful request.
- payment-service returned an error.
- inventory-service latency was 250 ms.
- payment-service version v2 was deployed.
- checkout-service called payment-service inside trace abc123.

Basic event shape:

{
  "tenant_id": "demo-company",
  "service_name": "payment-service",
  "event_type": "metric",
  "timestamp": "2026-05-19T10:00:00Z",
  "trace_id": "trace-123",
  "name": "http_request",
  "value": 1,
  "attributes": {
    "status_code": 500,
    "latency_ms": 1800,
    "endpoint": "/charge"
  }
}

Why tenant_id?

In real SaaS systems, one platform may serve many companies. tenant_id separates customer data.

Why service_name?

You need to know which service produced the signal.

Why trace_id?

Trace ID connects many service calls belonging to one user request.

Why attributes?

Different events need flexible extra details.

    name: str = Field(min_length=1)

    severity: str | None = None

    event_id: str


class Incident(BaseModel):

    tenant_id: str
    service_name: str
    title: str
    severity: str
    status: str
    summary: str

Create backend/app/storage.py:

from collections import defaultdict, deque
from datetime import datetime
from threading import Lock
from typing import Deque

from app.schemas import Incident, TelemetryEvent


class InMemoryStore:
    def __init__(self) -> None:
        self._lock = Lock()
        self.events: list[TelemetryEvent] = []
        self.incidents: list[Incident] = []
        self.recent_by_service: dict[str, Deque[TelemetryEvent]] = defaultdict(
            lambda: deque(maxlen=500)
        )

    def add_event(self, event: TelemetryEvent) -> None:
        key = f"{event.tenant_id}:{event.service_name}"
        with self._lock:
            self.events.append(event)
            self.recent_by_service[key].append(event)

    def get_recent_events(self, tenant_id: str, service_name: str) -> list[TelemetryEvent]:
        key = f"{tenant_id}:{service_name}"
        with self._lock:
            return list(self.recent_by_service[key])

    def add_incident(self, incident: Incident) -> None:
        with self._lock:
            self.incidents.append(incident)

    def list_incidents(self) -> list[Incident]:
        with self._lock:
            return list(reversed(self.incidents))


store = InMemoryStore()

7. Phase 2: Add Basic Anomaly Detection

Goal:

Detect when a service is unhealthy.

Simple rule:

- If last 20 request events have error rate above 20 percent, create an incident.
- If average latency is above 1000 ms, create an incident.

Create backend/app/anomaly.py:

from dataclasses import dataclass

from app.schemas import TelemetryEvent


@dataclass(frozen=True)
class AnomalyResult:
    is_anomaly: bool
    severity: str
    title: str
    evidence: list[str]


def analyze_service_window(events: list[TelemetryEvent]) -> AnomalyResult:
    request_events = [
        event
        for event in events
        if event.event_type == "metric" and event.name == "http_request"
    ]

    if len(request_events) < 20:
        return AnomalyResult(
            is_anomaly=False,
            severity="info",
            title="Not enough data",
            evidence=["Need at least 20 request events before detecting anomalies."],
        )

    last_events = request_events[-20:]

    error_count = 0
    latencies: list[float] = []

    for event in last_events:
        status_code = int(event.attributes.get("status_code", 200))
        latency_ms = float(event.attributes.get("latency_ms", 0))
        if status_code >= 500:
            error_count += 1
        if latency_ms > 0:
            latencies.append(latency_ms)

    error_rate = error_count / len(last_events)
    avg_latency = sum(latencies) / len(latencies) if latencies else 0

    evidence: list[str] = []

    if error_rate >= 0.20:
        evidence.append(f"Error rate is {error_rate:.0%} across the last 20 requests.")

    if avg_latency >= 1000:
        evidence.append(f"Average latency is {avg_latency:.0f} ms across the last 20 requests.")

    if evidence:
        severity = "critical" if error_rate >= 0.50 else "warning"
        return AnomalyResult(
            is_anomaly=True,
            severity=severity,
            title="Service health anomaly detected",
            evidence=evidence,
        )

    return AnomalyResult(
        is_anomaly=False,
        severity="info",
        title="Service healthy",
        evidence=[
            f"Error rate is {error_rate:.0%}.",
            f"Average latency is {avg_latency:.0f} ms.",
        ],
    )


Create backend/app/incident_engine.py:

from datetime import datetime, timezone
from uuid import uuid4

from app.anomaly import AnomalyResult
from app.schemas import Incident, TelemetryEvent
from app.storage import store


def maybe_create_incident(event: TelemetryEvent, anomaly: AnomalyResult) -> Incident | None:
    if not anomaly.is_anomaly:
        return None

    incident = Incident(
        id=str(uuid4()),
        tenant_id=event.tenant_id,
        service_name=event.service_name,
        title=f"{event.service_name}: {anomaly.title}",
        severity=anomaly.severity,
        status="investigating",
        summary=f"SignalForge detected abnormal behavior in {event.service_name}.",
        evidence=anomaly.evidence,
        created_at=datetime.now(timezone.utc),
    )
    store.add_incident(incident)
    return incident


Update backend/app/main.py:

from datetime import datetime, timezone
from uuid import uuid4

from fastapi import FastAPI

from app.anomaly import analyze_service_window
from app.incident_engine import maybe_create_incident
from app.schemas import IngestResponse, TelemetryEvent
from app.storage import store

app = FastAPI(title="SignalForge API", version="0.2.0")


@app.get("/health")
def health() -> dict[str, str]:
    return {"status": "ok"}


@app.post("/ingest", response_model=IngestResponse)
def ingest_event(event: TelemetryEvent) -> IngestResponse:
    event_id = str(uuid4())
    store.add_event(event)

    recent_events = store.get_recent_events(event.tenant_id, event.service_name)
    anomaly = analyze_service_window(recent_events)
    maybe_create_incident(event, anomaly)

    return IngestResponse(accepted=True, event_id=event_id)


@app.get("/events")
def list_events() -> dict:
    return {
        "count": len(store.events),
        "events": store.events[-50:],
        "server_time": datetime.now(timezone.utc),
    }


@app.get("/incidents")
def list_incidents() -> dict:
    incidents = store.list_incidents()
    return {"count": len(incidents), "incidents": incidents}


Interview explanation:

"Phase 2 adds the first intelligence layer. I calculate rolling service health from recent events. If error rate or latency crosses a threshold, the system creates an incident."


8. Phase 3: Build A Traffic Simulator

Goal:

Create fake services that send normal and broken traffic.

Create simulator/traffic_simulator.py:

import random
import time
from datetime import datetime, timezone

import requests


API_URL = "http://127.0.0.1:8000/ingest"
TENANT_ID = "demo-company"


def now_iso() -> str:
    return datetime.now(timezone.utc).isoformat()


def send_request_event(service_name: str, status_code: int, latency_ms: int) -> None:
    payload = {
        "tenant_id": TENANT_ID,
        "service_name": service_name,
        "event_type": "metric",
        "timestamp": now_iso(),
        "name": "http_request",
        "value": 1,
        "attributes": {
            "status_code": status_code,
            "latency_ms": latency_ms,
            "endpoint": "/demo",
        },
    }
    response = requests.post(API_URL, json=payload, timeout=5)
    response.raise_for_status()


def normal_traffic() -> None:
    services = ["checkout-service", "payment-service", "inventory-service"]
    service_name = random.choice(services)
    status_code = 200 if random.random() > 0.02 else 500
    latency_ms = random.randint(80, 250)
    send_request_event(service_name, status_code, latency_ms)


def bad_payment_traffic() -> None:
    status_code = 500 if random.random() < 0.45 else 200
    latency_ms = random.randint(900, 2200)
    send_request_event("payment-service", status_code, latency_ms)


if __name__ == "__main__":
    print("Starting simulator. Press Ctrl+C to stop.")
    print("First 30 events are normal. After that payment-service becomes unhealthy.")

    count = 0
    while True:
        if count < 30:
            normal_traffic()
        else:
            bad_payment_traffic()

        count += 1
        time.sleep(0.25)

10. Phase 5: Add Redis For Hot Rolling Windows

Goal:

Use Redis for fast recent counters.

Why?

PostgreSQL is good for durable storage, but rolling windows like "last 60 seconds error rate" are hot, frequently updated data. Redis is a better fit for that.

Install:

pip install redis

Add to backend/requirements.txt:

redis==5.0.8

Create backend/app/redis_window.py:

import json
import time

from redis import Redis

from app.schemas import TelemetryEvent


redis_client = Redis(host="localhost", port=6379, decode_responses=True)


def add_event_to_window(event: TelemetryEvent, window_seconds: int = 300) -> None:
    key = f"window:{event.tenant_id}:{event.service_name}"
    now = time.time()
    redis_client.zadd(key, {event.model_dump_json(): now})
    redis_client.zremrangebyscore(key, 0, now - window_seconds)
    redis_client.expire(key, window_seconds * 2)


def get_window_events(tenant_id: str, service_name: str, window_seconds: int = 300) -> list[TelemetryEvent]:
    key = f"window:{tenant_id}:{service_name}"
    now = time.time()
    raw_events = redis_client.zrangebyscore(key, now - window_seconds, now)
    return [TelemetryEvent.model_validate(json.loads(item)) for item in raw_events]


Use this instead of in-memory recent_by_service.

Interview explanation:

"Redis stores hot rolling windows because anomaly detection repeatedly asks for recent service events. PostgreSQL remains the durable source of truth, while Redis supports low-latency operational checks."

import type { Incident } from "./types";

const API_BASE_URL = "http://127.0.0.1:8000";

export async function fetchIncidents(): Promise<Incident[]> {
  const response = await fetch(`${API_BASE_URL}/incidents`);
  if (!response.ok) {
    throw new Error("Failed to fetch incidents");
  }
  const data = await response.json();
  return data.incidents;
}


Create frontend/src/App.tsx:

import { useQuery } from "@tanstack/react-query";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { fetchIncidents } from "./api";
import type { Incident } from "./types";
import "./App.css";

const queryClient = new QueryClient();

function IncidentCard({ incident }: { incident: Incident }) {
  return (
    <article className="incident-card">
      <div className="incident-header">
        <span className={`severity severity-${incident.severity}`}>
          {incident.severity}
        </span>
        <span>{incident.service_name}</span>
      </div>
      <h2>{incident.title}</h2>
      <p>{incident.summary}</p>
      <ul>
        {incident.evidence.map((item) => (
          <li key={item}>{item}</li>
        ))}
      </ul>
    </article>
  );
}

function Dashboard() {
  const { data, isLoading, error } = useQuery({
    queryKey: ["incidents"],
    queryFn: fetchIncidents,
    refetchInterval: 2000,
  });

  if (isLoading) {
    return <main className="page">Loading incidents...</main>;
  }

  if (error) {
    return <main className="page">Could not load incidents.</main>;
  }

  return (
    <main className="page">
      <section className="topbar">
        <div>
          <p className="eyebrow">SignalForge</p>
          <h1>Incident Triage Dashboard</h1>
        </div>
        <strong>{data?.length ?? 0} incidents</strong>
      </section>

      <section className="incident-grid">
        {data?.map((incident) => (
          <IncidentCard key={incident.id} incident={incident} />
        ))}
      </section>
    </main>
  );
}

export default function App() {
  return (
    <QueryClientProvider client={queryClient}>
      <Dashboard />
    </QueryClientProvider>
  );
}

from collections import Counter

from app.schemas import TelemetryEvent


def build_service_edges(events: list[TelemetryEvent]) -> list[dict]:
    edge_counts: Counter[tuple[str, str]] = Counter()

    for event in events:
        if event.event_type != "trace":
            continue

        caller = event.attributes.get("caller")
        callee = event.attributes.get("callee")

        if caller and callee:
            edge_counts[(caller, callee)] += 1

    return [
        {"source": source, "target": target, "count": count}
        for (source, target), count in edge_counts.items()
    ]

def find_recent_deployments(events, service_name, incident_time):
    matches = []
    for event in events:
        if event.event_type == "deployment" and event.service_name == service_name:

def keyword_search_runbooks(runbooks, query):
    query_terms = set(query.lower().split())
    scored = []
    for runbook in runbooks:
        text = f"{runbook.title} {runbook.body}".lower()
        score = sum(1 for term in query_terms if term in text)
        if score > 0:
            scored.append((score, runbook))
    return [runbook for score, runbook in sorted(scored, reverse=True)]

    vector = model.encode(text)
    return vector.tolist()


Search:

def search_similar_runbooks(db, tenant_id: str, query: str, limit: int = 5):
    query_embedding = embed_text(query)
    return (
        db.query(RunbookRow)
        .filter(RunbookRow.tenant_id == tenant_id)
        .order_by(RunbookRow.embedding.cosine_distance(query_embedding))
        .limit(limit)
        .all()

16. Phase 11: Add Root-Cause Ranking Without LLM First

Goal:

Build trustworthy ranking before adding generative AI.

Signals:

- error rate high: +30 points
- latency high: +20 points
- recent deployment: +25 points
- similar runbook found: +15 points
- dependency affected: +10 points

Create backend/app/triage.py:

from dataclasses import dataclass

from app.schemas import Incident, TelemetryEvent


@dataclass(frozen=True)
class Hypothesis:
    title: str
    confidence: float
    evidence: list[str]
    suggested_actions: list[str]


def generate_rule_based_hypotheses(
    incident: Incident,
    recent_events: list[TelemetryEvent],
) -> list[Hypothesis]:
    score = 0
    evidence: list[str] = []
    actions: list[str] = []

    has_recent_deployment = any(
        event.event_type == "deployment"
        and event.service_name == incident.service_name
        for event in recent_events
    )

    timeout_logs = [
        event
        for event in recent_events
        if event.message and "timeout" in event.message.lower()
    ]

    if incident.severity == "critical":
        score += 30
        evidence.append("Incident severity is critical.")

    if has_recent_deployment:
        score += 25
        evidence.append("A recent deployment happened for the affected service.")
        actions.append("Compare error rate before and after the deployment.")
        actions.append("Consider rolling back the latest deployment if no safer mitigation exists.")

    if timeout_logs:
        score += 20
        evidence.append(f"Found {len(timeout_logs)} timeout-related log events.")
        actions.append("Inspect downstream dependency timeout settings.")

    if not actions:
        actions.append("Inspect service logs and dependency health.")

    confidence = min(score / 100, 0.95)

    return [
        Hypothesis(
            title=f"{incident.service_name} regression or dependency timeout",
            confidence=confidence,
            evidence=evidence or incident.evidence,
            suggested_actions=actions,
        )
    ]

Interview explanation:

"I intentionally built a rule-based triage layer before the LLM. This gives a baseline that is explainable and testable. The LLM can later improve wording and combine evidence, but the core ranking does not depend entirely on it."

from fastapi import WebSocket

connected_clients: list[WebSocket] = []


@app.websocket("/ws/incidents")
async def incident_updates(websocket: WebSocket):
    await websocket.accept()
    connected_clients.append(websocket)
    try:
        while True:
            await websocket.receive_text()
    finally:
        connected_clients.remove(websocket)


async def broadcast_incident(incident):

19. Phase 14: Add Kafka Or Redpanda

Goal:

Make ingestion scalable.

Why not write directly to database?

During incidents, telemetry volume spikes. If every request writes directly to PostgreSQL, the database can become a bottleneck. Kafka/Redpanda decouples ingestion from processing.

New flow:

service -> ingestion API -> Kafka topic -> worker -> PostgreSQL/Redis/incidents

Topic design:

- telemetry.events
- telemetry.dead_letter
- incidents.created

Partition key:

tenant_id + service_name

Why?

Events for the same tenant and service stay ordered enough for rolling analysis.

Interview explanation:

"Kafka lets ingestion and processing scale independently. If workers are temporarily slow, events remain buffered. I use idempotent processing so retrying an event does not create duplicate incidents."
